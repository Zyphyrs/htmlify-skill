# htmlify — Any Content → Beautiful HTML Reference

Convert ANY input into a clean, tabbed, dark-theme HTML file for easy reading and reference.

## Trigger Conditions

Fire this skill when user:
- Pastes a README, handoff doc, or raw text and wants it visualized
- Gives a URL (GitHub repo, docs page, article) to convert
- Says anything like "gawing HTML", "parang yung report", "i-readable mo", "gawin mo reference"
- Gives a file path to convert

## Step 1: Parse Input

Detect what was given:
- **URL** → fetch the page/README with firecrawl_scrape or WebFetch
- **File path** → Read the file
- **Raw text / paste** → use as-is
- **GitHub repo URL** → fetch README.md via raw.githubusercontent.com or WebFetch

Extract:
- `TITLE` — name of the thing
- `CONTENT_TYPE` — readme / handoff / report / article / docs / mixed
- `SECTIONS` — identify logical sections in the content (features, setup, architecture, etc.)

## Step 2: Analyze Structure

Read the content and identify:
1. What is this about? (1-sentence summary)
2. What are the main sections? (map to HTML tabs)
3. Are there tables, code blocks, lists, diagrams? (preserve these)
4. What's the most important info? (goes in Overview tab first)

Plan tabs based on content. Common patterns:
- README → Overview / Features / Setup / Architecture / Config
- Handoff doc → Summary / What Was Built / Pending / Notes
- Report/article → Overview / Key Findings / Details / Sources
- Repo analysis → Overview / Details / Pros & Cons / Integration

Max 6 tabs. If content is simple, use fewer.

## Step 3: Generate HTML

Write a complete, self-contained HTML file using the design system below.

### Required HTML Structure

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{TITLE}</title>
  <style>/* full CSS here */</style>
</head>
<body>
  <!-- HEADER: title, subtitle, meta row (date, source, type badges) -->
  <div class="header">...</div>

  <!-- TABS: horizontal scrollable tab bar -->
  <div class="tabs">...</div>

  <!-- CONTENT: tab panels -->
  <div class="content">
    <!-- Panel 1: always Overview/Summary -->
    <!-- Panel N: based on content sections -->
  </div>

  <!-- FOOTER: source + generated by line -->

  <script>/* tab switcher */</script>
</body>
</html>
```

### Overview Tab (always first)

Must include:
- **Stats grid** (4-6 cards) — pull key numbers or facts from the content
- **What is this** card — 2-3 sentence summary
- **Key highlights** — top 3-5 bullets, most important info
- **Quick verdict / TL;DR** card if applicable

### Content Rules

- Convert markdown to proper HTML (headings → h2/h3, bullets → ul/li, code → `<code>` or `<pre>`)
- Tables → styled HTML tables with thead/tbody
- Code blocks → `<pre>` with syntax color (#a5b4fc text on var(--bg3) bg)
- Long text → break into readable paragraphs with proper spacing
- Badges for status/labels: use `.badge` classes (green/cyan/yellow/red/muted/indigo)
- Warnings → `.warn` box (yellow border), Notes → `.note` box (cyan border), Errors → `.danger` box (red border)
- Important callouts → `.success` box (green border)

### Tab Switcher Script (always include)
```js
function show(id, el) {
  document.querySelectorAll('.panel').forEach(p => p.classList.remove('active'));
  document.querySelectorAll('.tab').forEach(t => t.classList.remove('active'));
  document.getElementById(id).classList.add('active');
  el.classList.add('active');
  window.scrollTo({ top: 0, behavior: 'smooth' });
}
```

## Step 4: Save File

Output filename: `{slug}-reference.html`
- slug = lowercase, hyphens, no spaces (e.g. `trading-agents-reference.html`, `handoff-march-reference.html`)
- Save to current working directory
- Tell user the filename and that it's ready to open in browser

## Rules

- NEVER use emoji in the HTML content unless the source material has them
- NEVER truncate content — if it's long, add more tabs or use expandable sections
- ALWAYS include the footer: `Built by <a href="https://github.com/Zyphyrs">Zyphyrs</a> with Claude Code`
- Stats grid numbers should come from the actual content, not made up
- If content is a GitHub repo URL, always fetch the README first
- If content has no clear sections, use: Overview / Full Content / Key Notes
- Output ONLY the HTML file — no explanation needed unless user asks

## Gotchas

- Long READMEs: split into logical tabs, don't dump everything in one panel
- Handoff docs: prioritize "what's done" and "what's pending" as first two tabs
- Multiple files pasted together: treat as one document, find the structure
- URLs that 403/fail: ask user to paste the raw content instead
- Don't add EarlyVision context or any project context unless user explicitly provides it — this skill is generic
